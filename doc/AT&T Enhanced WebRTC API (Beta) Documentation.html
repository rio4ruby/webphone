Introduction

The AT&T Enhanced WebRTC (Web Real Time Communication) API allows you to build real time Web browser-based communication apps. The Enhanced WebRTC RESTful APIs allow you to discover other parties in communication and use a standard offer-and-answer model to exchange session parameters to establish effective audio-video communications.

There are three features that you can choose from within this API to shape the experience for your users:

  • AT&T Mobile Numbers:  AT&T mobile subscribers can make or receive voice or video calls from the app, using their existing AT&T mobile numbers.
  • Virtual Numbers:  A virtual number enables voice or video calls from an app or browser without requiring a physical SIM card or a physical device.
  • Account IDs: Call-enable the existing account IDs of your users that are associated to your domain name, giving them access to a rich set of communication features.

Note: A Premium Access account is required to use the Virtual Number and Account ID features.

Because the Enhanced WebRTC API provides access to the Public Switched Telephone Network and the ability to dial 911 from a WebRTC session, you must require the AT&T Mobile Number and Virtual Number users of your Enhanced WebRTC app to provide a physical address, in case emergency services need to be dispatched to their location. To accomplish this, AT&T provides the Emergency Services API, which creates an E911 identifier for your user’s address. The E911 identifier is associated with a user’s telephone number when creating a new WebRTC session. The E911 identifier associated with a WebRTC session should be updated when an AT&T Mobile Number or Virtual Number user changes locations.


Note: Emergency Services is not available for Account ID users.
API Quick Start
1
Get App Key & App Secret

i) To start working with AT&T APIs, create a developer account or sign into your existing account on the AT&T Developer Program website.

Click on the My Apps selection to set up a new app account by providing the Application Name, choosing Enhanced WebRTC, and selecting one or more of the following options.

  1. AT&T Mobile Number (displaying the user's AT&T phone number).
  2. Calling Access through existing member Account IDs (such as userID@yourdomain).
  3. Add new virtual telephone numbers to enable calling in my app.

ii) Add your company domain for the b and c options.

iii) The AT&T API Gateway is Cross-Origin Resource sharing (CORS) enabled. Register the domain where your application is hosted in the CORS domain list.

Applicable Scopes : WEBRTCMOBILE, EMERGENCYSERVICES, WEBRTC

Get Session Events

Select provider:

  • AT&T
Overview

The Get Session Events method retrieves the events during a specific session using a Comet LongPolling mechanism.
For more information about the Comet LongPolling mechanism, see rfc6202.
The events are related to call and conference.
Upon successful creation of a session, your app must call this method to send a request at regular intervals to maintain the event channel.
If no event channel is created, or if the event channel has expired, any event for that user is buffered and sent to your app as soon as the event channel is created.
When no event channel is created and the user ends the session, all events are discarded by the API Gateway.
Events are not stored and are not sent after a session is ended.
The following is the list of events under different scenarios.

  • Events during a set up or tear-down of a call
    • This event is received as a response for the Get Session Events method request when it is triggered from a remote party.
      The type for the event is call and the state indicates the current state of the call in the network.
      An event is generated under following circumstances.
      Note: The caller receives the event for all of the circumstances.
      • When the called party receives a call alert
      • When the called party rejects or ends a call
      • When the called party accepts or answers the call
      • When the caller cancels the call
      • When the caller ends the call
  • Events during ongoing call modifications
    • This event is generated when the customer initiates a media modification and should be expected under the following scenarios.
      Note: The destination receives the event for all of the circumstances.
      • Media upgrade or downgrade request, such as audio to audio-video
      • Call Hold Request
      • Call Resume Request
  • Events during modification completion
    • This event is an indication that the call modification is complete.
  • Events during call move complete
    • This event is an indication that the call move is complete.
  • Events during call transfer initiation
    • This event is an indication that the customer initiated a call transfer to a third-party and receives this event asynchronously on the event channel indicating that a request is sent to the API Gateway to bridge calls.
  • Events during call transfer completion

Stop Conference

Overview

The Stop Conference method rejects a conference invitation or ends an on-going conference.
The prerequisites for this method are as follows.

  • Your app must be registered and have a session identifier allocated.
  • To reject an invitation: Your app must receive a conference event with the state value invitation-received.
  • To end participation in a conference: Your app must receive a conference event with the state value session-open.
